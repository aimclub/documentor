# Описание пайплайна обработки DOCX документов

## Общая концепция

Пайплайн предназначен для полного извлечения структуры из DOCX документов с сохранением иерархии заголовков, текста, таблиц, изображений, подписей и элементов списка в правильном порядке.

**Ключевой принцип:** Используем DOTS OCR **только** для определения структуры (заголовки, подписи). Весь контент (текст, таблицы, изображения) извлекаем из XML DOCX, так как OCR часто даёт неполный или искажённый текст.

## Архитектура пайплайна

### Шаг 1: Конвертация DOCX → PDF

- DOCX конвертируется в PDF через `win32com` (COM-объект Word)
- PDF используется для Layout Detection

### Шаг 2: Layout Detection (DOTS OCR)

**Цель:** Определить структурные элементы документа и их позиции на страницах.

**Что делается:**
- PDF обрабатывается через DOTS OCR с увеличением 2x (`RENDER_SCALE=2`) для лучшего качества
- DOTS OCR определяет следующие типы элементов:
  - `Section-header` — заголовки разделов
  - `Caption` — подписи к таблицам и изображениям
  - `Image`/`Figure` — изображения
  - `Table` — таблицы
  - И другие типы (формулы и т.д.)
- Для каждого элемента сохраняется:
  - Bounding box (координаты на странице)
  - Номер страницы
  - Тип элемента

**Результат:** Списки section_headers, captions, ocr_images с их позициями.

### Шаг 3: Извлечение текста через Qwen OCR

**Цель:** Извлечь точный текст заголовков и подписей из изображений по bounding box.

**Что делается:**
- Для каждого `Section-header` и `Caption` из DOTS OCR:
  - Вырезается область изображения страницы по bounding box
  - Изображение отправляется в Qwen OCR для извлечения текста
- Фильтруются слишком короткие тексты (< 5 символов — скорее всего шум, номера страниц)

**Результат:** Списки headers_with_text, captions_with_text — заголовки и подписи с точным текстом.

### Шаг 4: Поиск заголовков в XML и построение правил

**Цель:** Сопоставить OCR-заголовки с элементами XML и построить правила для поиска пропущенных заголовков.

**Что делается:**

#### a) Извлечение всех элементов XML
- Парсится `word/document.xml` из DOCX архива
- Извлекаются все элементы в порядке появления:
  - Параграфы (paragraphs) — с полным текстом и свойствами форматирования
  - Таблицы (tables) — с полной структурой, включая объединённые ячейки
- Для параграфов с изображениями (`a:blip`) устанавливается флаг `has_image`
- Каждый элемент получает `xml_position` — порядковый номер в документе

#### b) Сопоставление OCR-заголовков с XML
- Для каждого OCR-заголовка ищется соответствующий параграф в XML через `find_header_in_xml_by_text`
- Поиск ведётся последовательно (от `last_found_xml_position + 1`), чтобы сохранить порядок
- Используется гибкое сопоставление:
  1. Точное совпадение текста
  2. Начало текста совпадает (минимум 5 символов для предотвращения ложных совпадений)
  3. Ключевые слова (Jaccard > 0.7) для длинных текстов
- Фильтруются элементы списка (если не нумерованный заголовок и не заголовок по стилю)
- Определяется уровень заголовка:
  - По стилю XML (style "1" = уровень 1, style "2" = уровень 2, ...)
  - По нумерации текста ("3.1." = уровень 2, "3.1.1." = уровень 3)

#### c) Построение правил для заголовков
- Анализируются свойства всех найденных заголовков для каждого уровня:
  - Шрифт (`font_name`)
  - Размер шрифта (`font_size`)
  - Жирность (`is_bold`)
  - Стиль (`style`)
  - Выравнивание (`alignment`)
- Правила используются для поиска пропущенных заголовков

#### d) Поиск пропущенных заголовков по правилам
- Для каждого параграфа XML, не отнесённого к заголовкам, проверяется соответствие правилам
- Приоритеты (веса):
  - **style_pattern** (стиль "1", "2", "3") — **вес 3.0** (максимальный)
  - Нумерованные паттерны ("1. Заголовок", "3.1. Подзаголовок")
  - Структурные ключевые слова ("Введение", "Заключение", "Список литературы")
  - Соответствие правилам (шрифт, размер, жирность, выравнивание)
  - Короткий жирный текст (`is_short_bold_text`) — потенциальный заголовок 3 уровня
- Текст, заканчивающийся на ":", **не** считается заголовком
- Подписи к таблицам/изображениям **не** считаются заголовками
- Динамические пороги:
  - `max_header_length` — на основе максимальной длины найденных заголовков (минимум 500)
  - `short_text_threshold` — 150 символов
  - `font_size_thresholds` — на основе статистики размеров шрифтов

#### e) Построение правил для подписей (captions)
- Аналогично заголовкам, анализируются свойства подписей
- Определяется общее выравнивание (часто "center" для подписей)

**Результат:** Полный список заголовков (OCR + правила), отсортированный по xml_position.

### Шаг 5: Извлечение таблиц и изображений из XML

**Что делается:**
- Извлекаются все таблицы из XML с полной структурой:
  - Ячейки с текстом
  - Объединённые ячейки (rowspan, colspan)
  - Количество строк и столбцов
- Извлекаются все изображения:
  - Путь к файлу в `media/` через relationships
  - Байты изображения
  - Размеры
- Первая таблица может быть пропущена (`--skip-first-table`) для документов с титульной страницей-таблицей

**Результат:** Списки docx_tables и docx_images с xml_position.

### Шаг 6: Сохранение изображений

**Что делается:**
- Изображения из media сохраняются с именами "Image X.png" (где X — порядковый номер)
- Сопоставляются с OCR-изображениями для определения номера страницы
- Создаётся `saved_images_map`: `{xml_position: {saved_name, page_num, image_index}}`

### Шаг 7: Построение иерархии (ключевой этап)

**Цель:** Построить полную структуру документа с правильной иерархией и порядком элементов.

**Ключевой принцип:** Обрабатываем ВСЕ элементы XML строго по порядку `xml_position`. Каждый непустой элемент добавляется в результат. Ничего не пропускаем.

**Предвычисление индексов:**
- `header_by_pos`: `{xml_position: header_data}` — из OCR + правила
- `tables_by_position`: `{xml_position: table_data}` — таблицы из XML
- `images_by_position`: `{xml_position: image_data}` — изображения из XML
- `properties_cache`: кэш свойств параграфов (чтобы не открывать ZIP повторно)
- `caption_positions_set`: множество позиций, уже обработанных как подписи

**Основной цикл:** Один проход по всем элементам `all_xml_elements`:

1. **Таблица (`type == 'table'`):**
   - Сохраняет текущий текстовый блок
   - Извлекает данные таблицы из `tables_by_position`
   - Если таблица не найдена в индексе — извлекает напрямую из XML элемента
   - Создаёт элемент `TABLE`

2. **Параграф (`type == 'paragraph'`):**
   
   a. **Изображение** (xml_pos в images_by_position):
   - Сохраняет текущий текстовый блок
   - Создаёт элемент `IMAGE` с именем файла и номером страницы
   
   b. **Пустой параграф** (без текста и без изображения):
   - Пропускается
   
   c. **Уже обработанная подпись** (xml_pos в caption_positions_set):
   - Пропускается
   
   d. **Заголовок из OCR/правил** (xml_pos в header_by_pos):
   - Фильтрует текст, заканчивающийся на ":" и подписи
   - Определяет уровень через `determine_header_level`
   - Создаёт элемент `HEADER_X` и обновляет стек заголовков
   
   e. **Заголовок по свойствам XML** (is_header_by_properties):
   - Проверяет: стиль "1","2","3"; is_heading_style; нумерованный паттерн; структурное ключевое слово; короткий жирный текст (≤100 символов)
   - Создаёт элемент `HEADER_X`
   
   f. **Подпись** (is_table_caption или is_image_caption):
   - Паттерны: "Таблица X", "Table X", "Рис. X", "Рисунок X", "Figure X"
   - Паттерны должны быть **в начале текста** (`^`)
   - Текст, заканчивающийся на ":", **не** считается подписью
   - Создаёт элемент `CAPTION`
   
   g. **Элемент списка** (is_list_item в свойствах):
   - Создаёт элемент `LIST_ITEM` с типом списка (numbered/bulleted)
   
   h. **Обычный текст:**
   - Добавляется в текущий текстовый блок
   - Текстовый блок сохраняется при:
     - Превышении MAX_TEXT_BLOCK_SIZE (3000 символов)
     - Превышении MAX_PARAGRAPHS_PER_BLOCK (10 параграфов)
     - Встрече следующего структурного элемента (заголовок, таблица, изображение, подпись, элемент списка)

**Определение уровня заголовка** (`determine_header_level`):
1. Если header_data содержит level — используем
2. Стиль = число ("1", "2", "3") — соответствующий уровень
3. Стиль Heading/заголовок — уровень из стиля
4. Структурные ключевые слова ("Введение", "Заключение") — уровень 1
5. Из нумерации текста:
   - "X" → уровень 1
   - "X.Y" → уровень 2
   - "X.Y.Z" → уровень 3
6. По контексту (стек заголовков):
   - Для ненумерованных: находим ближайший нумерованный родитель, уровень = его уровень + 1
   - **Важно:** Для соседних ненумерованных жирных заголовков уровень **не увеличивается** — они остаются на одном уровне (предотвращение каскада 3→4→5→6)

**Управление стеком заголовков:**
- `header_stack`: `[(level, element_id, is_numbered), ...]`
- При добавлении заголовка: удаляются из стека все заголовки с уровнем ≥ текущего
- `parent_id` = верхний элемент стека (или None)

**Результат:** Список `Element` в правильном порядке с корректными parent_id.

### Шаг 8: Сохранение результатов

**Выходные файлы:**
- `complete_structure.json` — полная структура документа (без OCR информации)
- `ocr_info.json` — отдельный файл с OCR-информацией (заголовки, подписи, их bbox и страницы)
- `images/` — папка с сохранёнными изображениями

## Известные ограничения и потенциальные проблемы

### 1. Зависимость от DOTS OCR для заголовков

**Проблема:** Если DOTS OCR не обнаруживает заголовок (Section-header), он может быть найден только через правила на основе свойств XML. Правила эффективны, когда заголовки имеют характерный стиль, нумерацию или форматирование, но могут пропустить нестандартные заголовки.

**Смягчение:** Правила проверяют стиль ("1", "2", "3"), нумерацию, структурные ключевые слова, жирность и размер шрифта. Стиль имеет максимальный вес (3.0).

### 2. Определение уровня для ненумерованных заголовков

**Проблема:** Для заголовков без нумерации (например, "Главная страница") уровень определяется по контексту (стеку заголовков). Это может быть неточным, если в документе нестандартная иерархия.

**Текущее решение:** Ненумерованные заголовки получают уровень = ближайший нумерованный родитель + 1. Соседние ненумерованные заголовки сохраняют один уровень.

### 3. Короткий жирный текст может быть ложно определён как заголовок

**Проблема:** Любой короткий (≤100 символов) жирный текст, не являющийся элементом списка, подписью или текстом с ":", может быть ложно определён как заголовок 3 уровня.

**Смягчение:** Дополнительные фильтры (не caption, не list_item, не заканчивается на ":").

### 4. Подписи определяются только по паттернам в начале текста

**Проблема:** Подписи определяются по регулярным выражениям ("Таблица X", "Рис. X"), которые должны быть в начале текста. Если формат подписей в документе отличается, они не будут обнаружены.

**Текущие паттерны:**
- Таблицы: `^таблица\s+\d+`, `^table\s+\d+`, `^табл\.\s*\d+`, `^tbl\.\s*\d+`
- Изображения: `^рис\.\s*\d+`, `^рисунок\s+\d+`, `^figure\s+\d+`, `^fig\.\s*\d+`, `^изображение\s+\d+`

### 5. Сопоставление OCR и XML элементов

**Проблема:** OCR-заголовки сопоставляются с XML последовательно по тексту. Если OCR даёт неточный текст (опечатки, пропуски), сопоставление может дать неправильный `xml_position`.

**Смягчение:** Гибкое сопоставление с порогом Jaccard > 0.7 и проверкой начала текста. Минимальная длина для fuzzy matching — 5 символов.

### 6. Таблицы без подписей

**Проблема:** Если перед таблицей нет подписи (или подпись в нестандартном формате), таблица всё равно добавляется, но связь "подпись → таблица" может быть утеряна.

### 7. Первая таблица в документе

**Проблема:** Некоторые DOCX документы имеют титульную страницу-таблицу (невидимую), которую нужно пропустить. Сейчас это делается через флаг `--skip-first-table` или автоматически для файлов с именем "Diplom2024".

### 8. Изображения без текста в параграфе

**Проблема:** Изображения в DOCX могут быть в параграфах без текста. Они определяются через `has_image` флаг и `a:blip` в XML. Если relationship файл повреждён, изображение может быть не найдено.

### 9. Определение жирности текста

**Проблема:** Жирность в DOCX определяется атрибутом `w:b` (или `w:bCs`). Если `val="false"` или `val="0"`, текст НЕ жирный. Если атрибут присутствует без значения — текст жирный. Эта логика реализована, но может быть сложной в нестандартных документах.

### 10. Универсальность пайплайна

**Проблема:** Пайплайн тестировался в основном на одном файле (Diplom2024.docx). Некоторые правила (например, стили "1", "2", "3" как заголовки) могут не работать для документов с другой стилизацией.

**Смягчение:** Правила строятся динамически на основе найденных OCR-заголовков. Стиль имеет максимальный вес, но если стили не используются, работают другие критерии (нумерация, ключевые слова, форматирование).

## Файлы в проекте

- `docx_complete_pipeline.py` — основной файл пайплайна (все функции)
