## Предисловие

В последний раз, при попытке сделать пайплайн с извлечением заголовков из docx мы столкнулись с некоторыми проблемами, а именно невозможность качественного извлечения заголовков из документа.

На это есть несколько причин:

- Не всегда заголовки отмечаются элементом HEADER 
- Качество семантического определения заголовка низкая и работает далеко не на всех документах
- Текст не всегда имеет разные шрифты, жирные выделения и другие признаки, которые могли бы указывать на заголовки

После выявления причин, было принято решение связаться с сотрудниками таких компаний как "Ростелеком" и "СберМед"

По итогам собеседований, исследования и мозгового штурма были выдвинуты два варианта реализации "определителя заголовков"

1) Использовать LLM и передавать ей скан изображения, с просьбой выделить заголовки
2) Использовать LLM и передавать ей текст с перекрытием, для определения заголовков

## Эксперименты по извлечению уровня заголовков

## Работа над линкингом структурных элементов

Планируется продумать метод хранения, линкования и создания иерархической связи элементов внутри документа.

На данный момент сложно давать ответ в каком виде это будет реализовано, но есть такая концепция:

Структурировать элементы по родительским и дочерним классам. Для этого нужно будет составить примерную структуру, и расставить как это должно быть. 
Например: у нас есть заголовок первого уровня, внутри него 2 заголовка второго уровня. В каждом заголовке есть таблицы и изображения с подписями.
Предполагается, что структура будет выглядеть так:
**Пример дерева**
```
Заголовок 1 уровня
│
├── Заголовок второго уровня N1
│   ├── Текст
│   ├── Таблица 1
│   │   └── Подпись к таблице 1
│   ├── Текст
│   └── Рисунок 1
│       └── Подпись к рисунку 1
│
└── Заголовок второго уровня N2
    ├── Текст
    ├── Таблица 2
    │   └── Подпись к таблице 2
    ├── Текст
    └── Рисунок 2
        └── Подпись к рисунку 2
```

Храним эту информацию в отдельном классе, какой элемент куда относится.

Хранение структуры позволяет сделать парсинг информации более гибким, например вместе с графиками и рисунками доставать пояснения.


## Работа над линкингом ссылок на структурные элементы внутри текста

Планируется улучшить пайплайн так, чтобы в текстовых полях, при ссылке на изображение (см. Рисунок 1, рис. 1 и т.д.) создавалась ссылка на сам элемент и его дочерние классы.

Возможные реализации:

1) Семантические через регулярные выражения. Возможные проблемы: сложно собрать набор из всех ссылок на нескольких языках. Например: рис. 1, изображение 1, график 1, фотография 1, диаграмма 1, схема 1 и так далее.

2) Создание ссылок с помощью LLM. Минусы - это долго и дорого.


## Интеграция с LangChain и архитектурные решения

### Переписывание библиотеки на LangChain

Планируется переписать библиотеку documentor на LangChain для улучшения интеграции и удобства использования. 

**Концепция использования:**
```python
from langchain import Document
from documentor import pipeline

document = Document(...)
result = pipeline(document)  # documentor pipeline работает с LangChain Document
```

### Работа с PDF через OCR

Для неразмечаемых PDF (где нельзя выделить текст):
1. Вызываем OCR pipeline для извлечения всего текста
2. Сохраняем структуру и связанность элементов

### Структурирование документа с помощью LLM

**Алгоритм работы с перекрытием:**

1. **Первый запрос:** Отправляем текст с перекрытием по параграфам в LLM
   - LLM определяет заголовки и их уровни
   - Устанавливает связи: заголовок 2 подвязан к заголовку 1
   - Определяет иерархию заголовков

2. **Последующие запросы:** Отправляем следующие части текста с перекрытием в один параграф
   - Передаём предыдущие заголовки для контекста
   - LLM определяет новые заголовки с учётом уже существующей структуры
   - Проверяет логику: внутри заголовка второго уровня не может быть заголовок первого уровня

3. **Построение графовой структуры:**
   - Каждому элементу присваиваем уникальный ID
   - Формат элемента: `(content: str, id: str, parent_id: str)`
   - Связываем элементы через `parent_id`

### Проблемы с изображениями в DOCX

**Выявленная проблема:**
В DOCX изображения могут парситься в неправильном порядке. Например:
- В документе 3 рисунка
- При вставке: сначала рисунок 3, потом рисунок 1, потом рисунок 2
- При парсинге первый рисунок будет третьим, что сбивает всю последовательность

**Возможное решение:**
- Использовать LLM для сравнения изображений: "это одинаковые картинки или нет?"
- Требует дополнительной проработки

### Поддержка различных форматов

1. **DOCX:**
   - Парсинг всего текста
   - Извлечение изображений (с учётом проблем с порядком)

2. **Markdown:**

   - LLM хорошо понимают Markdown

**Рассматривается возможность:**

Использовать одну LLM (например, QWEN VL) для всех задач:

**Вопросы для проработки:**

### Хранение изображений

**Формат хранения:**
- Подпись к изображению + само изображение
- Сохраняется в метаданных элемента

### Формат вывода пайплайна

После парсинга и разметки структуры документа, на выходе получаем:

**Формат:** список элементов, где каждый элемент содержит:
- `content: str` - текстовое содержимое
- `id: str` - уникальный идентификатор
- `parent_id: str` - идентификатор родительского элемента
- `metadata: dict` - метаданные (для изображений: изображение + подпись)

**Пример структуры:**
```python
[
    {
        "content": "Заголовок 1 уровня",
        "id": "h1_001",
        "parent_id": None,
        "metadata": {}
    },
    {
        "content": "Текст параграфа",
        "id": "p_001",
        "parent_id": "h1_001",
        "metadata": {}
    },
    {
        "content": "Подпись к рисунку 1",
        "id": "fig_caption_001",
        "parent_id": "h1_001",
        "metadata": {
            "image": <изображение>,
            "type": "figure"
        }
    }
]
```
