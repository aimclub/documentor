# Полное описание пайплайна с LangChain

## Введение

Documentor - opensource библиотека для предобработки данных перед RAG.

## Концепция использования

На вход подаём документ в виде `Document` из LangChain и получаем структурированный результат.

```python
from langchain_core.documents import Document
from documentor import pipeline

document = Document('path_to_file')
result = pipeline(document)
```

## Пример структуры вывода:
```python
[
    {
        "type": "header_1",
        "content": "Заголовок 1 уровня",
        "id": "00000001",
        "parent_id": None,
        "metadata": {}
    },
    {
        "type": "plain_text",
        "content": "Текст параграфа",
        "id": "00000002",
        "parent_id": "00000001",
        "metadata": {}
    },
    {
        "type": "header_2",
        "content": "Заголовок 2 уровня",
        "id": "00000003",
        "parent_id": "00000001",
        "metadata": {}
    },
    {
        "type": "picture",
        "content": "Рисунок 1 — Архитектура пайплайна",
        "id": "00000003",
        "parent_id": "00000001",

            "image_ref": "processed_documents/doc_.../images/img_0001.png",
            "mime_type": "image/png",
            "page_num": 5,

```

Примечания:
- `id` — уникальный идентификатор; каждый следующий элемент увеличивается на 1.
- `parent_id` — ссылка на родителя в иерархии (например, абзац относится к ближайшему заголовку).
- `metadata` — дополнительные поля (тип элемента, изображение, координаты, номер страницы, и т. д.).

## Поддерживаемые форматы (план)

- **PDF**
- **Markdown**

- TXT (в будущем)
- Excel (в будущем)

## Общее описание pipeline

1. **Вход**: пользователь передаёт документ/документы выбранных форматов в виде LangChain `Document`.
2. **Определение формата**: определяем исходный формат (PDF/Markdown/DOCX и т. д.).
3. **Выбор парсера**: по формату выбираем парсер и запускаем извлечение структуры и контента.
4. **Нормализация**: приводим результат разных парсеров к единому формату элементов (с `id`, `parent_id`, `metadata`, etc..).

## Описание парсеров

### Markdown

Markdown предполагается парсить без LLM.

План:
1) извлекать `page_content` из LangChain `Document`;
2) по регулярным выражениям определять элементы:
   - заголовки по количеству `#`;
   - таблицы по `| ... |` (и при необходимости учитывать выравнивание `---`);
   - списки, цитаты, код-блоки;
3) строить иерархию (заголовки → вложенные блоки).

### PDF

Для PDF сначала определяем, извлекается ли текст.

#### Выделяемый (извлекаемый) текст

1) если текст извлекается, используем `pdfplumber` (или аналог) для получения текста и базовой структуры;
2) Уточнения по пайплайну в процессе работы

#### Невыделяемый (скан) текст

Используем OCR-пайплайн:
1) определяем структуру страницы с помощью layout-модели;
2) выстраиваем последовательность элементов (порядок чтения);
3) распознаём текст OCR там, где это нужно (зоны текста/подписей);
4) собираем структурированный результат.

### DOCX

Для DOCX планируются два подхода (эксперименты):

1) **OCR-подход**: конвертировать страницы в изображения и извлекать структуру по визуальному представлению.
2) **Текстовый подход с перекрытием**: анализировать текст DOCX батчами (чанками) с перекрытием и восстанавливать структуру.

По результатам экспериментов выберем основной вариант или сделаем гибрид.

#### OCR-подход (черновой план)

1) рендерим страницы (или блоки) в изображения;
2) отправляем изображение в LLM и просим:
   - найти заголовки;
   - распознать/структурировать текст;
3) отдельно обрабатываем структурные элементы (изображения/таблицы/формулы) (нужно ещё подумать как).

Минусы:
- потенциально хуже качество на сложных документах;
- высокая стоимость и время выполнения.

#### Текстовый подход с перекрытием (черновой план)

1) отправляем текст чанками (например, ~3000 символов) с перекрытием (например, один абзац);
2) просим LLM:
   - найти заголовки;
   - классифицировать элементы (абзацы, подписи к изображениям и т. д.);
   - выстроить структуру (иерархию).

Риски:
- LLM может неверно восстановить структуру только по тексту;
- LLM может перепутать уровни заголовков.

Нюансы, которые нужно закрыть:
- ссылки на структурные элементы в тексте (например: «см. рис. 1», «см. табл. 2») — через регулярные выражения и/или LLM.

#### Структурные элементы (изображения/таблицы/формулы)

Здесь важна корректная привязка «подпись → объект» внутри текста и сохранение порядка. (ещё продумываем, по ходу работы)

- **Изображения**: возможна проблема с порядком. Например, в документе есть рис. 1, рис. 2, рис. 3, но внутри DOCX порядок вложений может не совпадать с визуальным порядком. Тогда можно получить неверное соответствие:
  - рис. 3 → рис. 1
  - рис. 1 → рис. 2
  - рис. 2 → рис. 3

  **Итог**: потеряна последовательность изображений и привязка подписей.

  **Требование**: решить задачу максимально эффективным способом; если не получится — придётся сопоставлять объекты по странице (визуально) и/или по контексту вокруг подписи.
- **Формулы**: нужно продумать извлечение формул (OMML/MathML, либо распознавание по изображению).
- **Таблицы**: нужно выбрать стратегию извлечения (структура, объединённые ячейки, подписи, экспорт в HTML/Markdown).

## Предложения по улучшению пайплайнов от ChatGPT 5.2 от которых я не отказался

1) **Единый контракт выходных данных**: зафиксировать набор обязательных полей (`id`, `parent_id`, `content`, `type`, `metadata`), чтобы все парсеры возвращали одинаковую схему.
2) **Порядок чтения (reading order)**: для layout/OCR явно хранить `page_num` и координаты блоков (bbox) и строить порядок чтения детерминированно.
3) **DOCX изображения**: привязку «подпись → изображение» делать по близости в документе (абзацы вокруг), по якорям/relationships, а при необходимости — по странице после рендера. (по-моему это так не работает)
4) **Идентификаторы**: сделать генерацию `id` детерминированной (например, по порядку + опционально по hash(source, page, bbox)), чтобы результаты были стабильны между прогонами.
5) **Оценка качества**: добавить минимальные метрики/тестовые корпуса (набор эталонных документов) и сравнение пайплайнов по структуре (precision/recall по заголовкам, привязкам подписей и т. д.).
